Customer tailored versions of Obsidian MEDILIG

Warning

The development release of the program, name of the vault, is ObsidianMediligDev.
Pay attention that this release should not be confused with the name of any customer vault.

The name of the vault for each customer will change with a unique name e.g. add the name of his company or his surname

Each customer will also write his own ToDo list with features that he would like to see implemented

Updates/changes should be applied only on Customer's System folder

For each new release that is distributed to the customer, include this Changelog and home pages such as Obsidian MEDILIG v0.9.1.en and Obsidian MEDILIG v0.9.1.el that will include the version of Obsidian MEDILIG.

Add/Change patient photo with the click of a button

A button was added above the photo of a patient to automate the process of changing the photo.

  1. New.Photo.tmpl
    new templater file is added with the code to handle the movement of the image file from the media_feeder to photo folder.
  2. Patient.Profile.type
    A button was added above the photo of a patient to automate the process of changing the photo by placing an image at Obsidian MEDILIG media_feeder. The image will be moved and placed under the photo folder that exists in patient's folder.

Modified

Moving media files from media_feeder to patient's assets folder

  1. New.Document.tmpl
    Code was modified to automate more the process of moving media files from a specific Windows folder, i.e. media_feeder, to patient's assets folder when the button new_document is pressed
    User will copy any files to the media_feeder and ObsidianMedilig will move them to assets and create the medical document
    folders inside media_feederare excluded when the movement takes place into assets

Simplify the creation of consultation and the linking to the encounter

  1. Modified Patient.Encounter.type: drop the consultation_link field from frontmatter
  2. Modified vw_patient_encounters.js: the code in current version is updated
  3. Modified vw_doctor_encounters.js: the code in current version is updated
  4. Modified consultation.comp.tmpl: metabind-js-view that binds the consultation filename to the encounter consultation_link field is deleted
  5. Modified patient.encounter.comp.tmpl: change dataviewjs code that displays the consultation date and time with the following: 
       const inlinks = dv.current().file.inlinks;
   const consPath = inlinks.find(link => link.path.includes("Consultation")).path;
   const pg = dv.page(consPath)
   const vdate = dv.func.dateformat(pg.date, "yyyy-MM-dd");
   const vdatetime = dv.date(`${vdate}T${pg.startTime}`); 
   const vdatetime_el = vdatetime.setLocale('el').toLocaleString(DateTime.DATETIME_HUGE); 
   dv.paragraph(dv.func.link(consPath, vdatetime_el));

v0.9.0 2024-07-23

Added

  1. Library of medical documents
  2. Medication in patient's encounter
  3. Table of contents in encounter and clinical case
  4. Buttons for quick access of med. history, medication, profile and med. library in encounter and clinical case
  5. List of medical documents at the last section of clinical case

Changed

  1. naming convention of files in folders
    Patient automatic numbering starts from 100 then auto increments.
    Clinical case numbering starts with 1 then auto increments
    Encounter numbering starts with 1 then auto increments
    Medical document numbering starts with 1 then auto increments
    Media files numbering start with 1 then auto increments
    Medical library, current medication, medical history and profile filenames include the patid
  2. For a new patient the clinical case must be completed first, then his first encounter. Consecutive encounters always belong to the current, most recent clinical case.

Fixed

When a new patient is created, the profile automatically completes fields
hname.first: INCOMPLETE
hname.last: INCOMPLETE
in case the record, patient folder is left idle without continuing its completion
