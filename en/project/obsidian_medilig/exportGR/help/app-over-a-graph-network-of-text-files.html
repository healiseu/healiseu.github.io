File over app — Steph Ango

This article discusses the "File over app" philosophy, which emphasizes the importance of creating digital content in formats that are durable and easily accessible over time. Here's a summary of the key points:

  1. The "File over app" philosophy advocates for creating digital artifacts in file formats that users can control and easily retrieve/read.
  2. It calls on tool makers to prioritize user data ownership, acknowledging that software is temporary.
  3. The article draws parallels with ancient artifacts like Egyptian hieroglyphs, emphasizing that the content (ideas) is more important than the tools used to create it.
  4. It criticizes the current trend of digital artifacts being stored in cloud services, databases, or proprietary formats, which can limit future access and control.
  5. The author suggests that for digital content to be readable in the far future (e.g., 2060s or 2160s), it should be compatible with older systems, even as far back as the 1960s.
  6. The importance of file durability is stressed not just for posterity but also for the creator's future access.
  7. The author mentions using Obsidian (an app they help develop) but acknowledges that the app itself will become obsolete. The focus is on the plain text files created, which are designed to last.
  8. The article concludes by stating that even if no one else reads these durable files in the future, creating them for one's future self is worthwhile.

This philosophy emphasizes long-term thinking in digital content creation and storage, prioritizing accessibility and control over short-term convenience or features tied to specific applications.

"App over a Graph Network of Markdown Files": A Progressive Approach to Digital Content Management

This philosophy proposes an advanced framework for digital content creation and management, building upon the foundational principles of the "file over app" approach while introducing sophisticated features typically associated with applications.

Core Components:

  1. Markdown as the Primary Format: Utilizing Markdown as the base file format ensures long-term readability and portability of content.
  2. Enhanced Metadata Structure: Incorporation of front-matter properties, tags, and both internal and external linking capabilities enriches the content ecosystem without compromising file integrity.
  3. Structured Asset Management: Media files and other assets are systematically organized in dedicated folders, maintaining a clear relationship with their referencing Markdown files.
  4. Automated Object Generation: Implementation of automated processes for creating interrelated entities (e.g., movies, actors, directors; patients, encounters, doctors; articles, posts, authors) streamlines content organization and relationship mapping.
  5. Dynamic Content Integration: Seamless combination of code, visualizations, and narrative text within single documents, facilitating comprehensive documentation of thought processes and enhancing result reproducibility.
  6. Interactive Elements: Integration of executable code cells, inline visualizations, and user input mechanisms (such as metadata fields and customizable buttons) transforms static documents into dynamic, interactive experiences.
  7. Reactive Content Display: Implementation of view fields that respond in real-time to changes in front-matter, providing dynamic content updates.
  8. Flexible Content Organization: Capability to organize entities into various views (grid, gallery, list, kanban), offering versatile content presentation options.

This approach aims to bridge the gap between traditional file systems and full-fledged applications, preserving the durability and accessibility of file-based systems while incorporating the interactive and organizational capabilities typically associated with dedicated software applications.

By leveraging the strengths of both paradigms, this philosophy enables the creation of a robust, flexible, and future-proof digital content ecosystem. It is particularly well-suited for personal information management, research documentation, and complex project organization, offering a balanced solution that prioritizes both long-term data integrity and advanced functionality.

Let's delve deeper into the contrast between traditional app architecture and your "App over a Graph Network of Markdown Files" philosophy:

Traditional App Architecture:

  1. Separation of Concerns:
    • Front-end: Handles user interface and user interactions
    • Back-end: Manages data storage, processing, and business logic
    • Database: Stores structured data separately from the application logic
  2. Data Flow:
    • Data typically flows from the database through the back-end API to the front-end
    • User interactions on the front-end trigger requests to the back-end, which then queries or updates the database
  3. Data Storage:
    • Data is stored in structured databases (e.g., SQL, NoSQL)
    • Clear separation between data and presentation layer
  4. Scalability:
    • Can scale front-end and back-end independently
    • Allows for distributed systems and load balancing
  5. Development:
    • Often requires different skill sets for front-end and back-end development
    • Changes to data structure may require updates in multiple layers

"App over a Graph Network of Markdown Files" Philosophy:

  1. Unified Structure:
    • Each Markdown file acts as both data storage and presentation layer
    • Front-matter contains structured data (resembling database records)
    • File content includes both data and interactive elements
  2. Data Flow:
    • Data and interactions are localized within each file
    • Inter-file connections create a graph-like structure, allowing for complex relationships without a separate database
  3. Data Storage:
    • Data is stored directly within Markdown files
    • Front-matter properties act as structured data fields
    • File content serves as both data and presentation
  4. Scalability:
    • Scales horizontally by adding more files
    • May face challenges with very large datasets or high concurrency
  5. Development:
    • Potentially simpler development process, as changes to data and presentation occur in the same file
    • Requires a different mindset, blending content creation with application development
